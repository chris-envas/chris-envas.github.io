【每日一问】this究竟指向哪里

发表于 2019-12-26 | 更新于 2020-06-20

困惑

以上是两个最典型的困惑，this的行为和预期的并不一致，this究竟指向哪里？

解惑

this是在函数在被调用的那一刻发生的绑定，函数被调用的那一刻所处的位置决定了this的指向

要找到函数调用的位置，首先需要分析调用栈，什么是调用栈？

也许我们可以先从栈讲起，在计算机数据结构中它是一种基本的线性数据结构，特点是后进先出，入栈操作为push,出栈操作为pop，数组就是一个典型的栈！

当我们在浏览器执行脚本时，每一个执行的函数都会遵照特定的顺序被推入栈里，而这个栈就是调用栈！它存储了当前需要执行的函数，JS的解释器依赖于它来追踪函数的执行顺序，通过它我们可以追踪到哪个函数正在执行，执行的函数体中又调用了哪些函数，实际上我们可以把它想象成函数调用链，下面一起来进行一次调用栈的分析

执行步骤

1、console.log(bar(6)) 函数被调用

2、bar函数被调用

3、foo函数被调用

4、foo函数执行完毕，弹出栈

5、bar函数执行完毕，弹出栈

6、console.log(bar(6))函数执行完毕，弹出栈

上述流程，如图所示

可能会有人好奇main函数是哪里来的，实际上我们运行程序时，我们首先查找main函数–所有的其他函数都是在main函数中执行,这是JS解释器层面设定的问题，无需挂怀！

到这一步，笔者认为你一定已经深刻的认识到调用栈的含义，通过调用栈我们可以知道当前正在执行的函数，而调用位置就是当前正在执行的函数的前一个调用中，理解这句话是至关重要的！this的绑定发生在首次函数运行的那一刻，首次执行的函数的前一个调用就是this指向的区域！牢记！

找打调用位置后再依据函数的上下文对象,判断this最终指向！

默认绑定

上下文对象：默认window(浏览器中非严格模式)

工作流程：

1、执行bar函数

2、该函数被添加进调用栈中

从调用栈中我们可以发现bar函数的前一个调用为main,说明此时this的指向全局！

隐式绑定

上下文对象：函数的引用对象

工作流程：

1、执行bar()函数

2、该函数被添加进调用栈中

调用栈与上面的例子一样，可是为什么打印的结果却是2？

这是因为此刻bar函数有一个上下文对象， 调用位置会使用obj来引用函数，当函数的引用有上下文对象时this将绑定到这个上下文对象中

工作流程：

1、执行foo()函数

2、该函数被添加进调用栈中

OK,从调用栈中，很明显可以知道foo位于栈顶，所有它的this应该指向全局，不过等等，仔细看看该函数是否有上下文对象，没有！到这里我们可以百分百的肯定，this指向全局！

这种行为被称为隐式绑定丢失，实际上，笔者并不关心这些所谓的概念，因为调用栈已经给了你答案

显示绑定

上下文对象：call或appl传递的参数对象

遇到call或apply等关键词就无需分析调用栈了，因为bar.call会强行设置上下文对象，将this的指向绑定到新的上下文对象,这也是绝大多数开发者惯用的手段，帮助我们更快使代码达到预期的效果！

new绑定

上下文对象：被执行构造行为的函数对象

关于new的来源及工作原理，可以参考new关键字解析 )

简单的讲使用new来调用函数，会指向下面的操作

1、创建一个全新的对象

2、将这个新对象的原型进行连接

3、并将函数的this指向绑定到新对象中

4、如果函数没有返回其他对象，new 表示式的函数会调用自动返回这个新对象

其实在new的底层实现中，肯定也是借助了call、apply关键词来实现this的绑定

不论是call、apply、还是bind,甚至是new，它们的工作原理都是是强行的为当前的函数绑定一个新的上下文对象

this指向受控于函数调用位置和函数上下文对象的影响
